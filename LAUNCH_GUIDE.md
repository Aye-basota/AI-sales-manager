# AI Sales Manager — пошаговая инструкция по запуску

Эта инструкция поможет запустить проект с нуля до первой отправленной кампании, даже если вы не программист. Все команды можно скопировать и вставить в командную строку.

**Что мы сделаем:**
1. Подготовим внешние аккаунты (Telegram, OpenRouter).
2. Запустим систему в Docker.
3. Добавим живой Telegram-аккаунт, с которого будут идти сообщения.
4. Создадим первый скрипт продаж.
5. Загрузим 1 тестовый контакт и запустим кампанию через Telegram Admin Bot.

> **Важно:** на момент написания гида в проекте 377 тестов проходят, 0 критических багов, но есть 3 некритичных ограничения (см. раздел «Известные ограничения» внизу). Инструкция честно их отражает.

## 0. Что понадобится до начала

| Что нужно | Зачем | Где взять |
|---|---|---|
| Docker + Docker Compose | Запускает все части системы (базу, бота, планировщик) одной командой | [docker.com](https://www.docker.com/products/docker-desktop/) |
| Telegram API ID и API Hash | Чтобы программа могла работать от имени живого Telegram-аккаунта | [my.telegram.org](https://my.telegram.org) |
| Токен Telegram-бота (Admin Bot) | Управление кампаниями через Telegram | Напишите [@BotFather](https://t.me/BotFather), создайте бота и скопируйте токен |
| OpenRouter API Key | Генерация сообщений искусственным интеллектом | [openrouter.ai/keys](https://openrouter.ai/keys) |
| Живой Telegram-аккаунт (номер телефона) | С него будут отправляться сообщения клиентам | Любой реальный аккаунт, желательно «возрастной» (не свежезарегистрированный) |
| Telegram User ID тестового контакта | Чтобы отправить первое сообщение | Напишите [@userinfobot](https://t.me/userinfobot), он ответит вашим ID |

**Несколько терминов простыми словами:**

- **MTProto** — протокол, через который работают официальные приложения Telegram. Проект использует его, поэтому сообщения идут от живого пользователя, а не от бота.
- **Session string** — длинная строка-ключ, которая позволяет программе отправлять сообщения от вашего Telegram-аккаунта без повторного ввода SMS-кода.
- **Docker Compose** — инструмент, который поднимает сразу несколько «мини-компьютеров» (контейнеров): базу данных PostgreSQL, кэш Redis, API-сервер, планировщик и Telegram-бота.
- **Seller-аккаунт** — живой Telegram-аккаунт, который отправляет сообщения потенциальным клиентам.

## 1. Установка Docker и Docker Compose

1. Скачайте и установите **Docker Desktop** по ссылке выше.
2. Убедитесь, что Docker запущен:

```bash
docker --version
docker-compose --version
```

Если обе команды вывели номера версий — всё хорошо.

**Если не сработало:**
- На Windows запускайте терминал от имени обычного пользователя (не обязательно администратор), но Docker Desktop должен быть включён.
- Если пишет `command not found` — перезагрузите компьютер после установки Docker.

## 2. Получаем внешние ключи

### 2.1 Telegram API ID / API Hash

1. Войдите на [my.telegram.org](https://my.telegram.org) под тем Telegram-номером, который будет seller-аккаунтом.
2. Перейдите в **API development tools**.
3. Создайте приложение (можно любое название, например `AISales`).
4. Скопируйте **api_id** (число) и **api_hash** (строка из букв и цифр).

### 2.2 Токен Admin Bot

1. В Telegram найдите [@BotFather](https://t.me/BotFather).
2. Отправьте `/newbot`, придумайте имя и адрес (username) для бота.
3. BotFather пришлёт токен вида `123456789:ABCdefGHI...`. Сохраните его.

### 2.3 Ключ OpenRouter

1. Зарегистрируйтесь на [openrouter.ai](https://openrouter.ai).
2. Перейдите в раздел Keys и создайте ключ.
3. Скопируйте ключ вида `sk-or-v1-...`.

### 2.4 ID тестового контакта

1. В Telegram найдите [@userinfobot](https://t.me/userinfobot).
2. Нажмите **Start** — бот пришлёт ваш ID (число, например `123456789`).
3. Запишите это число. Его мы используем для первого тестового контакта.

## 3. Копируем проект и настраиваем окружение

### 3.1 Клонируем репозиторий

```bash
git clone <ссылка-на-репозиторий>
cd ai-sales-manager
```

(Если у вас просто архив — распакуйте и перейдите в папку проекта.)

### 3.2 Создаём файл `.env`

```bash
cp .env.example .env
```

> На Windows в PowerShell: `Copy-Item .env.example .env`

Откройте `.env` в любом текстовом редакторе и заполните:

```dotenv
DATABASE_URL=postgresql+asyncpg://sales:salespass@postgres:5432/ai_sales
REDIS_URL=redis://redis:6379/0
OPENROUTER_API_KEY=sk-or-v1-ВАШ_КЛЮЧ_OPENROUTER
ADMIN_BOT_TOKEN=ВАШ_ТОКЕН_ОТ_BOTFATHER
ADMIN_NOTIFICATION_CHAT_ID=
SECRET_KEY=придумайте_любую_длинную_строку
SESSION_ENCRYPTION_KEY=
TELEGRAM_API_ID=ВАШ_API_ID_ЧИСЛОМ
TELEGRAM_API_HASH=ВАШ_API_HASH
DAILY_MESSAGE_LIMIT=50
DEBUG=True
```

**Что обязательно заполнить:**
- `OPENROUTER_API_KEY`
- `ADMIN_BOT_TOKEN`
- `TELEGRAM_API_ID`
- `TELEGRAM_API_HASH`

**Что желательно заполнить:**
- `ADMIN_NOTIFICATION_CHAT_ID` — ID чата/пользователя, куда приходить уведомления о горячих лидах. Можно оставить пустым, но тогда алёртов не будет.
- `SESSION_ENCRYPTION_KEY` — если оставить пустым, session-строки аккаунтов будут храниться в базе в открытом виде. Чтобы сгенерировать ключ, выполните:

```bash
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

> Если `python` не найден — используйте `python3`. На Windows убедитесь, что Python добавлен в PATH.

Скопируйте полученную строку (вида `gAAAAAB...`) в `SESSION_ENCRYPTION_KEY`.

## 4. Запускаем систему

### 4.1 Собираем и запускаем контейнеры

```bash
docker-compose up -d --build
```

Это может занять несколько минут при первом запуске.

### 4.2 Проверяем, что всё поднялось

```bash
docker-compose ps
```

Вы должны увидеть 3 сервиса со статусом `Up`:
- `ai-sales-db` (PostgreSQL)
- `ai-sales-redis` (Redis)
- `ai-sales-api`

> **Почему только 3?** Шедулер кампаний, Admin Bot и слушатели входящих сообщений запускаются внутри контейнера `ai-sales-api` (см. `app/main.py`). Раньше в `docker-compose.yml` были отдельные сервисы `ai-sales-scheduler` и `ai-sales-admin-bot`, но они не имели точек входа и бесконечно рестартовали. Они убраны, чтобы Docker запускался чисто.

### 4.3 Применяем миграции базы данных

```bash
docker-compose exec api alembic upgrade head
```

Если всё прошло успешно, в конце будет что-то вроде:

```
INFO  [alembic.runtime.migration] Running upgrade  -> 9fbbddc0c495, Initial migration
```

**Если не сработало:**
- `connection refused` — подождите 30 секунд и повторите. PostgreSQL мог ещё не успеть запуститься.
- `relation already exists` — база уже была инициализирована раньше. Это не страшно.

### 4.4 Проверяем API

Откройте в браузере:

```
http://localhost:8000/health
```

Должен появиться ответ `{"status":"ok"}` или похожий.

Или в терминале:

```bash
curl http://localhost:8000/health
```

## 5. Добавляем seller-аккаунт (живой Telegram)

Сейчас система работает, но ещё не от чьего имени отправлять сообщения. Нужно авторизовать живой Telegram-аккаунт и получить **session string**.

### 5.1 Генерируем session string

Самый простой способ — запустить временный Python-скрипт внутри контейнера `api`.

Выполните:

```bash
docker-compose run --rm api bash
```

Вы попадёте внутрь контейнера. Там выполните:

```python
python - <<'PY'
import asyncio
import os
from pyrogram import Client

async def main():
    api_id = int(os.getenv("TELEGRAM_API_ID"))
    api_hash = os.getenv("TELEGRAM_API_HASH")
    async with Client("session_gen", api_id=api_id, api_hash=api_hash, in_memory=True) as app:
        session = await app.export_session_string()
        print("\n=== ВАША SESSION STRING ===")
        print(session)
        print("===========================\n")

asyncio.run(main())
PY
```

Скрипт спросит:
1. Номер телефона — введите номер seller-аккаунта в международном формате (`+79991234567`).
2. Код из SMS или Telegram — введите полученный код.
3. Если включена двухфакторная аутентификация — введите пароль.

В конце появится длинная строка — это `session string`. **Скопируйте её полностью** и выйдите из контейнера (`exit`).

**Если код не приходит:**
- Убедитесь, что ввели номер правильно и с `+`.
- Попробуйте получить код через Telegram вместо SMS (на том устройстве, где установлен аккаунт).
- Если аккаунт новый, Telegram может временно блокировать авторизацию сторонних приложений — используйте более «возрастной» аккаунт.

### 5.2 Добавляем аккаунт в систему

Откройте в браузере Swagger (интерактивную документацию API):

```
http://localhost:8000/docs
```

1. Найдите раздел `telegram-accounts`.
2. Разверните запрос `POST /telegram-accounts` и нажмите **Try it out**.
3. Вставьте JSON (замените значения на свои):

```json
{
  "phone": "+79991234567",
  "session_string": "ВАША_SESSION_STRING_ИЗ_ПРЕДЫДУЩЕГО_ШАГА",
  "username": "your_seller_username",
  "status": "ready"
}
```

4. Нажмите **Execute**.

Если всё ок, в ответе появится созданный аккаунт с полем `"status": "ready"`.

**Альтернатива через командную строку** (если Swagger не открывается):

```bash
curl -X POST "http://localhost:8000/telegram-accounts" \
  -H "Content-Type: application/json" \
  -d '{
    "phone": "+79991234567",
    "session_string": "ВАША_SESSION_STRING",
    "username": "your_seller_username",
    "status": "ready"
  }'
```

> **Почему `status: "ready"`?** Система отправляет сообщения только аккаунтами со статусом `ready` или `active`. Если оставить `warming` (статус по умолчанию), сообщения не пойдут.

### 5.3 Проверяем логи

Чтобы убедиться, что аккаунт подключился и слушает входящие:

```bash
docker-compose logs -f api
```

Ищите строки вида:

```
Inbound listener started for account ...
```

Если их нет — перезапустите контейнеры:

```bash
docker-compose restart
```

## 6. Создаём первый скрипт продаж

Скрипт — это инструкция для ИИ: кем он является, кому пишет, какая цель и в каком тоне.

1. В Telegram найдите бота, созданного в BotFather, и нажмите **Start**.
2. Отправьте `/newscript`.
3. Бот пошагово спросит:

| Параметр | Рекомендуемое значение для теста | Пояснение |
|---|---|---|
| Название | `Тестовый скрипт` | Любое понятное имя |
| Роль (role_prompt) | `Ты менеджер по продажам IT-решений для малого бизнеса.` | Кто пишет |
| Целевая аудитория | `-` | Можно пропустить, отправив `-` |
| Цель (goal) | `Договориться о коротком созвоне` | Чего хотим добиться |
| Критерий успеха | `-` | Можно пропустить |
| Тон | `Деловой` | Выберите кнопкой |
| Max messages | `2` | Сколько сообщений может отправить система одному контакту |
| Follow-up delay | `24` | Через сколько часов отправить напоминание, если не ответили |
| Рабочие часы | `📝 Указать вручную` | Для теста укажите время, которое включает текущий момент |

После ввода рабочих часов и таймзоны бот покажет сводку. Нажмите **✅ Создать**.

**Совет для теста:** если сейчас 20:00, укажите рабочие часы `09:00 - 23:00` и таймзону `Europe/Moscow`. Иначе система подождёт завтрашнего 09:00.

## 7. Готовим тестовый контакт

Создайте файл `test_contact.csv` в папке проекта. Содержимое:

```csv
telegram_username,telegram_user_id,first_name,company_name,position
mytestuser,123456789,Иван,ООО Тест,Директор
```

Замените:
- `mytestuser` на username тестового контакта **без @**.
- `123456789` на реальный Telegram User ID из [@userinfobot](https://t.me/userinfobot).
- `Иван`, `ООО Тест`, `Директор` — на любые тестовые данные.

> **Важно:** в таблице обязательно должна быть колонка `telegram_user_id`. Без числового ID система пропустит контакт и не отправит сообщение.

## 8. Загружаем контакт и запускаем кампанию

1. В Telegram отправьте боту команду `/upload`.
2. Бот попросит файл — отправьте `test_contact.csv`.
3. Бот покажет превью и кнопки. Нажмите **✅ Создать кампанию**.
4. Выберите созданный скрипт.
5. Введите название кампании, например `Тестовая кампания`.
6. Нажмите **▶️ Запустить**.

Бот ответит:

```
Кампания Тестовая кампания запущена с 1 контактами.
```

Система сразу пытается отправить первое сообщение. Если сейчас рабочие часы и аккаунт готов — сообщение уйдёт в течение минуты.

### Проверяем, что сообщение отправилось

1. Посмотрите логи:

```bash
docker-compose logs -f api
```

Ищите строки без ошибок и что-то вроде `process_campaigns` / `sent`.

2. Откройте Telegram от имени тестового контакта — там должно появиться первое сообщение от seller-аккаунта.

3. Можно также посмотреть статус кампании в боте: `/campaigns`.

**Если сообщение не пришло сразу:**
- Подождите до 5 минут — планировщик запускается каждые 5 минут.
- Проверьте логи `api`.
- Убедитесь, что текущее время попадает в рабочие часы скрипта.

## 9. Проверяем ответы (inbound)

Если тестовый контакт ответит на сообщение, система:

1. Сохранит входящее сообщение.
2. Определит намерение (положительный ответ, возражение, согласие на созвон и т.д.).
3. Сгенерирует и отправит ответ.

Смотрите логи:

```bash
docker-compose logs -f api
```

Если ответ классифицируется как `meeting_intent` или `hot`, и вы заполнили `ADMIN_NOTIFICATION_CHAT_ID`, вам придёт алёрт в Telegram.

## Типичные ошибки и решения

### «No eligible account for contact» в логах

**Причина:** нет аккаунта со статусом `ready`/`active`, у которого есть `session_string` и который не превысил дневной лимит.

**Решение:**
- Проверьте, что создали аккаунт через `/telegram-accounts` со статусом `ready`.
- Проверьте, что `session_string` не пустая.
- Проверьте `daily_messages_sent` — если больше 50, аккаунт ушёл в cooldown до полуночи.

### «Skipping contact (no telegram_user_id)»

**Причина:** в CSV не было колонки `telegram_user_id` или она была пустая.

**Решение:** добавьте колонку `telegram_user_id` с числовым ID.

### Admin Bot не отвечает на команды

**Причина:**
- Неправильный `ADMIN_BOT_TOKEN`.
- Где-то ещё запущен второй экземпляр бота с этим токеном (Telegram позволяет только один polling).

**Решение:**
- Проверьте `.env`.
- Выполните `docker-compose restart api` (Admin Bot работает внутри контейнера `api`).
- Убедитесь, что вы пишете именно тому боту, которого создавали.

### FloodWait / PeerFlood в логах

**Причина:** Telegram ограничил аккаунт за слишком активную рассылку.

**Решение:**
- Подождите (обычно от нескольких минут до суток).
- Для production аккаунт нужно «прогревать» несколько дней: добавлять аватар, bio, подписываться на каналы, писать друзьям. В проекте есть вспомогательный скрипт `scripts/warmup_accounts.py`, который только меняет статус; реальный прогрев делается вручную.
- Используйте несколько seller-аккаунтов — система будет ротировать их.

### Ошибка портов: «bind: address already in use»

**Причина:** на компьютере уже запущены PostgreSQL или Redis на портах 5432 / 6379.

**Решение:** остановите локальные службы или измените порты в `docker-compose.yml` (но это уже задача для разработчика).

### Миграции не применяются

**Причина:** контейнер `postgres` ещё не готов.

**Решение:** подождите и повторите `docker-compose exec api alembic upgrade head`.

## Известные ограничения (честно)

Проект работает, но у него есть несколько нюансов. Они не мешают тестовому запуску с 1 контактом, но стоит о них знать.

1. **Flood от входящих сообщений.** Если лид напишет много сообщений подряд, аккаунт может ответить на все и уйти в cooldown. В B2B такое редко, но имейте в виду.
2. **Race condition при очень быстрых ответах.** Теоретически, если два входящих сообщения придут одновременно, возможен двойной ответ. На практике вероятность низкая.
3. **Аналитика считает «ответы» и по неактивным кампаниям.** Цифры в `/analytics` могут включать активность из paused/closed кампаний. Это косметический недочёт.
4. **`processed_contacts` считает сообщения, а не контакты.** Если одному контакту отправлено initial + follow-up, счётчик покажет `2/1`. Это не влияет на реальную отправку.
5. **Для production аккаунт нужно прогревать.** Свежезарегистрированный номер сразу уйдёт в бан. Используйте «живые» аккаунты с историей.
