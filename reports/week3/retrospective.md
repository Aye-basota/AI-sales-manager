# Sprint Retrospective

## What went well

1. **The MVP v1 scope stayed focused** — the team kept the Sprint centered on the core Telegram outreach flow instead of expanding the backlog too early.
2. **Traceability improved** — the issue-based backlog, stable user-story IDs, and linked report artifacts made it easier to connect scope, implementation, and evidence.
3. **The architecture supports future changes** — the funnel model and multi-provider LLM design isolated product-specific changes from the broader orchestration code.

## What did not go well

1. **Backlog and evidence setup took manual effort** — creating and maintaining the repository workflow artifacts required more coordination than the implementation work itself.
2. **Some alignment gaps appeared late** — a few details across schema, prompts, and documentation needed rework once the MVP v1 shape became clearer.
3. **Public demonstration packaging still needs discipline** — screenshots, release links, and video evidence are easy to postpone unless they are planned as first-class Sprint work.

## Action points

1. Add a short release-evidence checklist to Sprint planning so screenshots, demo links, and report updates are prepared before the final review.
2. Reserve a dedicated refinement step for schema/API alignment and acceptance-criteria review before implementation starts.
