# AI Sales Manager — дорожная карта доработок

> Промежуточный результат зафиксирован в git. Ниже собраны наблюдения и предложения по улучшению, которые стоит реализовать в следующих итерациях.

---

## 1. Human-like отправка сообщений

### 1.1. Слишком долгий "печатает..."
**Проблема:** сейчас typing-индикатор держится всё время рассчитанной задержки. Для длинных сообщений это выглядит неестественно (десятки секунд).

**Что сделать:**
- Ввести разумный потолок для typing-задержки (например, 8–12 секунд).
- Периодически "прерывать" печать, чтобы имитировать естественные паузы человека.
- Добавить случайный "thinking" перерыв перед ответом, но не показывать typing всё это время.

### 1.2. Сообщения всё ещё приходят одним пластом
**Проблема:** `split_message_into_chunks` уже есть, но для типичных ответов chunk получается один. Пользователь видит одно большое сообщение.

**Что сделать:**
- Жёстче ограничивать максимальный размер одного чанка (например, 300–400 символов).
- Разбивать не только по пустым строкам, но и по предложениям, если абзац слишком длинный.
- Добавить заметную паузу между чанками (2–6 секунд).
- Опционально: отправлять короткое вводное сообщение первым, а развёрнутую мысль — вторым.

## 2. Воронка продаж и стиль первого контакта

### 2.1. Как работает воронка сейчас
1. **Campaign (кампания)** — привязана к одному `Script` и содержит список `Contact` через `CampaignContact`.
2. **Scheduler** раз в 5 минут обходит `running` кампании и для каждого контакта в статусе `pending`/`initial_sent`:
   - проверяет рабочие часы скрипта;
   - создаёт `Conversation`;
   - генерирует первое outbound-сообщение через LLM;
   - отправляет его и переводит `CampaignContact` в `initial_sent`.
3. **Inbound listener** слушает ответы. Если лид отвечает:
   - классифицирует намерение (`meeting_intent`, `question`, `objection` и т.д.);
   - генерирует ответ;
   - меняет `Conversation.current_state` через state machine (`cold → warm → hot → meeting_booked` / `closed`).
4. **Follow-up** отправляется только через `follow_up_delay_hours`, если контакт не ответил.

### 2.2. Первое сообщение сразу предлагает звонок
**Проблема:** даже при `tone: friendly` system prompt жёстко требует "получить согласие на 15-минутный звонок" и "предложить 2 варианта времени". Из-за этого первое сообщение выглядит агрессивно.

**Что сделать:**
- Разделить скрипт на этапы воронки (hook → qualification → value → call-to-action).
- Первое сообщение должно быть мягким hook'ом (1–2 предложения), без прямого приглашения на звонок.
- Предложение созвона появляется только после положительного ответа / вовлечённости.
- Добавить поле в `Script`: `first_message_goal` или `sales_stage`, чтобы можно было выбирать стиль первого контакта.

## 3. Telegram Admin Bot — UX/UI

### 3.1. Меню и авто-комплит команд
**Проблема:** команды (`/newscript`, `/upload`, `/campaigns` и т.д.) приходится вводить вручную. Новый пользователь не помнит список.

**Что сделать:**
- Добавить постоянную клавиатуру (reply keyboard) с основными действиями.
- Регистрировать команды через `BotCommand` (`/setcommands`), чтобы Telegram показывал подсказки и авто-комплит.
- В начале диалога (`/start`) выводить интерактивное меню с кнопками.

### 3.2. Наглядная структура: скрипты, кампании, контакты
**Проблема:** непонятно, как связаны сущности.

**Что сделать:**
- В `/start` или `/help` добавить схему:
  ```
  Script (сценарий общения)
    ↓
  Campaign (рассылка по списку контактов)
    ↓
  Contact (человек) → Conversation (диалог)
  ```
- В списке кампаний показывать название скрипта и количество контактов.
- В списке скриптов показывать, сколько кампаний их использует.

### 3.3. Просмотр диалога по горячему лиду
**Проблема:** `/hotleads` показывает только последние 1–2 сообщения. Нельзя посмотреть полную историю.

**Что сделать:**
- Добавить inline-кнопку "📜 История диалога" под каждым hot lead.
- При нажатии выгружать последние N сообщений (например, 20) с временными метками.
- Опционально: экспорт диалога в txt/csv.

### 3.4. `/upload` — несоответствие описания
**Проблема:** в инструкции и LAUNCH_GUIDE написано, что нужна колонка `telegram_user_id`, но в интерфейсе бота и в подсказках это не отражено / названо `telegram_id`.

**Что сделать:**
- При `/upload` показывать пример CSV с обязательными колонками.
- Поддерживать оба варианта названия: `telegram_user_id` и `telegram_id` (маппинг).
- При ошибке импорта явно писать: "Не найдена колонка telegram_user_id (или telegram_id)".
