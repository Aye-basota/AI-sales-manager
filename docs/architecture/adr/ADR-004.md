# ADR-004: Anti-Repetition Check for Generated Messages

## Status

Accepted

## Context

AI Sales Manager may generate several messages within the same conversation (initial outreach, follow-up, objection handling). LLMs tend to reuse phrasing, especially with similar prompts and short context windows. Sending the same or near-identical text twice to a lead:

- Signals automation and reduces trust
- Increases the risk of Telegram spam filters flagging the account
- Violates the product goal of human-like, varied sales dialogues

Customer feedback on MVP v1 highlighted that repetitive outbound text makes conversations feel robotic. A repetition check must run automatically on every generated message before send, without operator intervention.

Alternatives considered:

1. **Higher LLM temperature only** — reduces but does not eliminate repetition; non-deterministic and hard to test.
2. **Embedding-based similarity** — more semantically accurate, but adds model/API dependency and latency on every check.
3. **Store-and-compare last N messages with string similarity** — lightweight, deterministic, testable.

## Decision

Implement anti-repetition in `app/llm/guardrails.py` as `check_anti_repetition()`:

- Compare the candidate message against the **last 5 outbound messages** in the same conversation.
- Use Python's `difflib.SequenceMatcher` ratio as the similarity metric.
- Reject messages with similarity **≥ 0.8** (configurable threshold parameter).
- Integrate into `evaluate_guardrails()` — a repetition violation causes `apply_guardrails()` to return `None`, triggering LLM retry with stricter instructions (same pipeline as [ADR-001](ADR-001.md)).

The last-message list is supplied by the caller (scheduler or inbound handler) from conversation history already loaded for context.

## Consequences and Tradeoffs

**Positive:**

- Repetitive outputs are blocked before Telegram delivery, supporting [QR-04](../../quality-requirements.md#qr-04) user error protection.
- SequenceMatcher is fast, has no external dependencies, and supports boundary-value unit tests at the 0.8 threshold ([QRT-04](../../quality-requirement-tests.md#qrt-04-anti-repetition-in-generated-messages)).
- Lives alongside other guardrails checks, so one call site enforces all pre-send policies.

**Negative:**

- Lexical similarity ≠ semantic similarity — paraphrased repeats with different wording may pass; structurally different messages on the same topic may fail.
- The 0.8 threshold is a heuristic; legitimate follow-ups that restate a call-to-action may be rejected and require LLM retry.
- Comparing only the last 5 messages does not detect repetition against older history in long threads.

**Neutral:**

- Tuning the threshold or window size requires updating tests and possibly revisiting this ADR if product feedback demands it.

## Quality Requirements Addressed

- [QR-04: Anti-Repetition in Generated Messages](../../quality-requirements.md#qr-04) — User Error Protection (Usability): blocks near-duplicate outbound text before send.

## Related Architecture Views

- **Static view:** anti-repetition is a sub-responsibility of the Guardrails component ([ADR-001](ADR-001.md)).
- **Dynamic view:** LLM generate → guardrails (including repetition check) → send or retry loop.
- **Deployment view:** in-process check; conversation history read from PostgreSQL before each generation.
