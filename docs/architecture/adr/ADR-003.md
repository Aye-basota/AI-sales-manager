# ADR-003: Scheduler-Driven Outbound Processing

## Status

Accepted

## Context

Outbound campaigns must send initial messages and follow-ups to many contacts across multiple Telegram accounts while respecting:

- Working hours and timezone per script
- Follow-up delay (default 24 h) and auto-close (48 h)
- Per-account daily message limits and rate limits (1 message / 30 s)
- Account cooldown after FloodWait / PeerFlood errors

A purely event-driven approach (send only when a user starts a campaign) would miss timed follow-ups and recovery after restarts. A tight loop or cron per campaign would not scale and would complicate persistence across deploys.

The scheduler interval is **5 minutes** ([QR-03](../../quality-requirements.md#qr-03)). Processing must finish within that window for typical campaign sizes so jobs do not stack and duplicate sends.

Alternatives considered:

1. **Message queue workers (Celery/RQ)** — better for very high throughput, but adds Redis queue semantics and worker processes beyond current MVP scope.
2. **Per-campaign asyncio tasks** — hard to persist and recover after process restart.
3. **Manual operator-triggered sends only** — unacceptable for autonomous follow-up and auto-close requirements.

## Decision

Use **APScheduler** (`AsyncIOScheduler`) with a **SQLAlchemyJobStore** backed by PostgreSQL so job definitions survive application restarts.

Core design in `app/core/scheduler.py`:

- `process_campaigns()` runs every **5 minutes** via `IntervalTrigger(minutes=5)`.
- Only campaigns with `status == "running"` are processed.
- For each pending contact, the scheduler checks working hours, follow-up delay, account eligibility (`ready`/`active`, valid session, cooldown expired), and daily limits.
- Account selection rotates across eligible accounts; LLM generation and Pyrogram dispatch run in the same async job.
- Supporting jobs: `reset_daily_counters` (cron midnight Europe/Moscow), `recover_cooldown_accounts` and `auto_close_conversations` (every 6 hours).
- Campaign start via Admin Bot or API can trigger an immediate `process_campaigns()` run in addition to the periodic tick.

FloodWait and PeerFlood errors move accounts to cooldown rather than failing the entire batch.

## Consequences and Tradeoffs

**Positive:**

- Bounded 5-minute processing cycle aligns with [QR-03](../../quality-requirements.md#qr-03) time-behaviour scenarios.
- SQLAlchemyJobStore gives crash-safe scheduling without a separate job broker.
- Centralized `process_campaigns()` keeps anti-spam, working-hours, and account-rotation logic in one place — easier to test ([QRT-03](../../quality-requirement-tests.md#qrt-03-outbound-message-delivery-performance)).
- Immediate processing on campaign start improves perceived responsiveness while the periodic job handles follow-ups.

**Negative:**

- Worst-case latency for a new pending contact is up to one scheduler interval (~5 min) unless manually triggered.
- Heavy LLM latency for large batches can approach the interval boundary; guardrail retries ([ADR-001](ADR-001.md)) add further delay.
- APScheduler requires a synchronous DB URL for the job store while the app uses async SQLAlchemy elsewhere — an intentional dual-driver compromise.

**Neutral:**

- Scheduler correctness depends on accurate server time and timezone configuration in scripts and Docker environment.

## Quality Requirements Addressed

- [QR-03: Outbound Message Delivery Performance](../../quality-requirements.md#qr-03) — Time Behaviour (Performance Efficiency): bounded scheduler cycle for account selection, generation, and dispatch.

## Related Architecture Views

- **Static view:** Scheduler orchestrates Campaign Service, Account Pool, LLM Engine, Guardrails, and SellerClient (Pyrogram).
- **Dynamic view:** primary outbound sequence diagram — scheduler tick through message delivery.
- **Deployment view:** scheduler runs inside the main application container; job state stored in PostgreSQL; Redis used for caching elsewhere in the stack.
