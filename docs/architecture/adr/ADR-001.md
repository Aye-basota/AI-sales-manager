# ADR-001: LLM Output Guardrails

## Status

Accepted

## Context

AI Sales Manager sends outbound Telegram messages through live user accounts (Pyrogram MTProto), not the Bot API. The LLM layer (`app/llm/engine.py`) generates message text on behalf of sales personas. Generated text can contain:

- Off-brand or legally sensitive topics (politics, religion, hate speech)
- Formatting that looks automated (markdown, emoji)
- Script or language mismatches (CJK, Arabic characters in Russian sales dialogs)
- Self-identification as a bot or AI assistant

If such content reaches `SellerClient.send_message()`, it can damage the client's brand, expose legal liability, or trigger Telegram account restrictions. The guardrail check must run **before** dispatch and must be deterministic enough to verify in automated tests.

Alternatives considered:

1. **Prompt-only constraints** — cheaper to maintain, but LLMs still violate instructions unpredictably.
2. **Post-send moderation** — too late; the message is already delivered.
3. **Human review for every message** — does not scale for campaign throughput.

## Decision

Introduce a dedicated guardrails module (`app/llm/guardrails.py`) that evaluates every LLM output through `evaluate_guardrails()` / `apply_guardrails()` before sending. The module checks:

- Message length (max 300 words)
- Forbidden topic keywords
- Bot/AI self-identification phrases (whole-word matching)
- Markdown symbols (`#`, `*`, `_`, `` ` ``)
- Emoji characters
- CJK and Arabic script ranges

When any check fails, `apply_guardrails()` returns `None`. The LLM engine retries generation with a stricter prompt; if guardrails fail again, a safe fallback text is used instead of sending unsafe content.

The guardrails layer sits between the LLM engine and the outbound message pipeline (scheduler / inbound handler), keeping Telegram dispatch free of policy-violating text.

## Consequences and Tradeoffs

**Positive:**

- Unsafe or off-brand content is blocked before it reaches Telegram, directly supporting [QR-01](../../quality-requirements.md#qr-01).
- Checks are pure functions with no I/O — fast, deterministic, and easy to unit-test ([QRT-01](../../quality-requirement-tests.md#qrt-01-llm-response-safety)).
- Violation types are logged via `GuardrailsResult.violations`, which helps tune thresholds without changing dispatch code.

**Negative:**

- Keyword-based topic blocking can reject legitimate messages that mention a forbidden word in an acceptable context.
- Strict no-emoji and no-markdown rules may reduce conversational naturalness.
- Guardrail rejection triggers LLM retries, adding latency and API cost on the outbound critical path ([ADR-003](ADR-003.md)).

**Neutral:**

- Anti-repetition is implemented in the same module but documented separately in [ADR-004](ADR-004.md) because it addresses a distinct quality concern ([QR-04](../../quality-requirements.md#qr-04)).

## Quality Requirements Addressed

- [QR-01: LLM Response Safety](../../quality-requirements.md#qr-01) — Confidentiality (Security): blocks forbidden content before delivery.

## Related Architecture Views

- **Static view:** guardrails module between LLM Engine and Message Dispatcher components.
- **Dynamic view:** outbound campaign flow — LLM generate → `apply_guardrails()` → send or retry.
- **Deployment view:** guardrails run in-process inside the application container; no separate moderation service.
