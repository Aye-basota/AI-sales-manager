# ADR-007: Production Monitoring and Logging

## Status

Accepted

## Context

After MVP v1 deployment, the team had no structured way to know whether the API, scheduler, PostgreSQL, or Redis were healthy. Container restarts were manual, and logs were inconsistent. MVP v2 must be production-ready enough to self-report health and survive routine failures.

## Decision

1. Keep the existing `/health` endpoint (checks DB + scheduler) as the primary uptime signal.
2. Add centralized logging via `app/logging_config.py` that writes structured plain-text logs to stderr, controlled by the `LOG_LEVEL` environment variable.
3. Add a Docker Compose `healthcheck` to the `api` service and set `restart: unless-stopped` for `postgres`, `redis`, and `api`.

The health endpoint is intentionally lightweight so it can be polled every 30 seconds by UptimeRobot or similar services.

## Consequences

- Positive: failures are visible through standard container logs and uptime checks.
- Positive: Docker restarts unhealthy/failed containers automatically.
- Negative: the health endpoint does not verify Telegram or LLM provider connectivity; those are checked indirectly through scheduler logs and outbound metrics.
