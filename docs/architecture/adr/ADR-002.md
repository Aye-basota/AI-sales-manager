# ADR-002: Deterministic Conversation State Machine

## Status

Accepted

## Context

The sales funnel tracks each lead through stages such as `cold`, `warm`, `hot`, `follow_up`, `objection_handler`, `meeting_booked`, and `closed`. Campaign scheduling, follow-up timing, hot-lead alerts, and analytics all depend on the current conversation state.

Without explicit transition rules, state updates could be scattered across handlers (inbound bot, scheduler, admin API), leading to:

- Closed or booked leads receiving further outbound messages
- Incorrect funnel metrics (e.g., counting a `meeting_booked` lead as still `warm`)
- Hard-to-reason-about edge cases when multiple events arrive close together

Alternatives considered:

1. **Free-form status strings updated ad hoc** — flexible initially, but error-prone as features grow.
2. **Database triggers** — centralizes rules but hides logic from application tests and code review.
3. **External workflow engine** — adds operational complexity for a bounded set of states.

## Decision

Implement a deterministic, table-driven state machine in `app/core/state_machine.py`:

- States and events are typed literals (`State`, `Event`).
- All allowed transitions live in `_TRANSITIONS: dict[State, dict[Event, State]]`.
- `transition(current_state, event)` returns the next state; undefined events leave the state unchanged.
- Terminal states (`meeting_booked`, `closed`) have empty transition maps — they never exit.
- `is_terminal(state)` is used by schedulers and handlers to skip further outbound messaging.

Inbound handlers classify user replies into events (`positive_reply`, `negative_reply`, `meeting_intent`, etc.) and call `transition()` once per event. The state machine module has no database or network dependencies.

## Consequences and Tradeoffs

**Positive:**

- Transition behaviour is fully specified in one module, supporting [QR-02](../../quality-requirements.md#qr-02) fault tolerance.
- Terminal-state immutability prevents messaging closed or booked leads — a common source of customer-facing defects.
- Pure functions enable exhaustive unit tests with 100% branch coverage ([QRT-02](../../quality-requirement-tests.md#qrt-02-conversation-state-correctness)).
- New states require an explicit row in `_TRANSITIONS`, making funnel changes reviewable in PRs.

**Negative:**

- Every new funnel stage or event type requires updating the transition table update plus tests — no runtime configuration.
- `informational` events preserve state, which is correct but can surprise developers expecting a visible state change.
- The state machine does not persist history; audit trails rely on conversation records elsewhere.

**Neutral:**

- Funnel stage progression (`app/core/funnel.py`) complements but does not replace conversation state — both must stay aligned when the product evolves.

## Quality Requirements Addressed

- [QR-02: Conversation State Correctness](../../quality-requirements.md#qr-02) — Fault Tolerance (Reliability): deterministic transitions and immutable terminal states.

## Related Architecture Views

- **Static view:** `State Machine` component consumed by Inbound Handler, Scheduler, and Analytics.
- **Dynamic view:** inbound reply flow — classify event → `transition()` → persist state → conditional outbound.
- **Deployment view:** runs in-process; state persisted in PostgreSQL via the ORM layer.
